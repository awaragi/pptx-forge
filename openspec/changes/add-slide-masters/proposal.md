## Why

pptxgenjs supports real PowerPoint slide masters (`pptx.defineSlideMaster()` / `pptx.addSlide({ masterName })`), but nothing in `src/lib/` exposes them today — every slide instead redraws its own chrome (border, header, footer) by hand via `frame.js`, called from every slide file. Masters are core PowerPoint functionality, not an optional add-on, and slide authors currently have no way to select between different chrome/background treatments per slide (e.g. a title slide vs. a content slide) without duplicating draw calls. This change adds masters as a first-class `lib` capability, and makes workspaces able to define their own master variants the same way they already override `theme.js`.

## What Changes

- Add `src/lib/masters.js`, exporting:
  - a default export that is a **factory function**, `(theme) => SlideMasterProps[]`, returning plain pptxgenjs `SlideMasterProps` objects (`{ title, background?, objects?, slideNumber?, margin? }` — `title` mandatory and unique, matching what pptxgenjs itself requires). A function is unavoidable: master content needs `theme` values (colors, radius, grid coordinates) to stay consistent with workspace overrides, and a plain static object has no way to reference them. Critically, this function is only ever called **once**, at `createLib` time, when `theme` is already fully resolved — everything downstream of that (merging, registering on `pptx`) works with the resulting plain array and never re-invokes it.
  - the library ships exactly **one** default master: `{ title: 'BLANK', objects: [] }` — a proof that the wiring works end-to-end, not an opinionated chrome implementation. `frame.js`'s existing per-slide chrome pattern is unchanged and untouched by this change.
  - `mergeMastersByTitle(defaults, overrides)` — merges two already-generated `SlideMasterProps[]` arrays by `title`: an override entry whose `title` matches a default replaces that default entry wholesale; an entry with a new `title` is appended. This operates on plain data, with no knowledge of `theme` or functions.
  - `applyMasters(pptx, masterDefinitions)` — takes the already-merged plain array and calls `pptx.defineSlideMaster(entry)` once per entry, unmodified. It does not merge and does not need `theme` — both already happened by the time it runs.
- `createLib(themeOverrides, masterOverrides)` gains a second, optional parameter: `masterOverrides`, a factory function of the same `(theme) => SlideMasterProps[]` shape as the library default (or `undefined`/absent). `createLib` resolves `theme` (as it already does), calls the default factory and (if present) `masterOverrides` with that resolved `theme`, merges the two results via `mergeMastersByTitle` **once**, and exposes both `lib.masters` (a plain array of the merged titles, e.g. `['BLANK']`, for discovery) and `lib.masterDefinitions` (the merged plain array itself, for the orchestrator to hand to `applyMasters` later without recomputing anything).
- `bin/forge.js` and `src/tools/browser/compile.js` (the two orchestrators) each gain one new step: after `pptx` is constructed and before the slide loop runs, call `applyMasters(pptx, lib.masterDefinitions)` — reusing the array `createLib` already generated, not calling any factory function again.
- Workspaces may add an optional `workspaces/<slug>/masters.js`, default-exporting the same `(theme) => SlideMasterProps[]` factory shape, to add new masters or override existing ones by `title`. The compile pipeline treats it as optional (absence handled the same way a missing `theme.js` is handled today).
- **New scaffolding file**: `src/sample/masters.js`, a commented-out example (mirroring `src/sample/theme.js`'s existing convention) demonstrating the factory-function shape and theme-constant usage. `bin/create.js` copies it into every new workspace alongside `theme.js` and `slides/deck.js`.
- `INSTRUCTIONS.md` — the reference an AI reads unconditionally, unlike the opt-in `COMPONENTS.md` companion — documents `masters.js`: the `(theme) => SlideMasterProps[]` factory shape, referencing `theme` constants (colors, radius, grid, header/footer text) rather than hardcoding values, and `pptx.addSlide({ masterName: '<title>' })` (a literal string — `lib.masters` is for discovery only) as the encouraged default for `addSlide` calls once a workspace has masters beyond `BLANK`. This is what "core, not optional" means here: the *capability* is always documented, never gated behind asking for `COMPONENTS.md` the way `frame`/`comp`/`layout` are. It does **not** mean every workspace must have a `masters.js` file — that file stays optional at the workspace level, exactly like `theme.js` is today; a workspace with neither still compiles, using only the library's `BLANK` default. `INSTRUCTIONS.md` makes no reference to `frame`/`comp`/`layout` themselves, which remain scoped entirely to `COMPONENTS.md`. `COMPONENTS.md` gains a note that manual `frame.border`/`slideHeader`/`slideFooter` chrome remains available as a one-off alternative to a dedicated master.
- Placeholders (`{ placeholder: ... }`) are technically supported — nothing in `applyMasters` filters or rejects them, since each merged entry is passed straight through to `pptx.defineSlideMaster()` unmodified — but `INSTRUCTIONS.md`/`COMPONENTS.md` don't document or encourage them. They're PowerPoint-editable regions meant for a human to fill in after export, which doesn't fit this library's generate-once model, so they're simply left out of the authoring guidance rather than actively excluded.
- `comp`/`layout`/`prim`/`tables` are not usable inside a master's factory function at all — masters are plain pptxgenjs object literals, parameterized only by `theme`. (`tables`/`addMedia` were never possible regardless — pptxgenjs's master `objects` union has no `table`/`media` variant.)
- **`pptx-forge.html`'s browser tool treats `masters.js` exactly like `theme.js`** — a permanent, non-deletable sidebar entry, pre-populated from `src/sample/masters.js` (mirroring how the existing `theme.js` placeholder is sourced from `src/sample/theme.js`), visually muted until edited away from that placeholder, excluded from the alphabetical slide-file sort, with Discard/Rename unavailable and a Reset-to-placeholder action shown in their place. This touches five existing browser-tool capabilities:
  - `browser-forge`: the sidebar/toolbar/Reset behavior above, plus `Forge` treating `masters.js`'s default export as the `masterOverrides` factory (like `theme.js`'s export is treated as `themeOverrides`) rather than executing it as a slide module.
  - `browser-workspace-export-import`: exported/imported `.zip` archives include `masters.js` alongside `theme.js` and slide files.
  - `browser-slide-transfer`: Move/Copy actions are unavailable for `masters.js`, same as `theme.js`.
  - `browser-workspace-persistence`: `masters.js` content is persisted to `localStorage` and restored the same way `theme.js` is; the default blank workspace state includes a placeholder `masters.js` alongside placeholder `theme.js`.

## Capabilities

### New Capabilities
- `lib-slide-masters`: `src/lib/masters.js` — the default master factory, `mergeMastersByTitle`, `applyMasters`, and the workspace override contract.

### Modified Capabilities
- `lib-api-contract`: `createLib` signature changes from `createLib(overrides)` to `createLib(themeOverrides, masterOverrides)`, where `masterOverrides` is an optional `(theme) => SlideMasterProps[]` factory function. The returned object gains two members: `masters` (a plain array of registered title strings) and `masterDefinitions` (the merged plain `SlideMasterProps[]`, for orchestrator use).
- `lib-module-split`: the enumerated `src/lib/` file list gains `masters.js`.
- `slide-authoring-guide`: `INSTRUCTIONS.md` must document the workspace `masters.js` factory-function shape, theme-constant usage, and `addSlide({ masterName: '<title>' })` as the encouraged pattern — without referencing `frame`/`comp`/`layout`, which stay scoped to `COMPONENTS.md`.
- `components-authoring-guide`: `COMPONENTS.md`'s `lib.frame` documentation gains a note that manual `frame.*` chrome is a one-off alternative to a dedicated master.
- `workspace-scaffold`: `bin/create.js` also copies `src/sample/masters.js` into every new workspace, alongside `theme.js` and `slides/deck.js`.
- `browser-forge`: `pptx-forge.html` treats `masters.js` as a permanent sidebar entry with placeholder/muted/Reset behavior mirroring `theme.js`, and `Forge` passes its default export to `createLib` as `masterOverrides`.
- `browser-workspace-export-import`: exported/imported workspace `.zip` archives include `masters.js`.
- `browser-slide-transfer`: Move/Copy are unavailable for `masters.js`, same as `theme.js`.
- `browser-workspace-persistence`: `masters.js` is persisted/restored per-workspace the same way `theme.js` is, and the default blank-workspace state includes a placeholder `masters.js`.

## Impact

- New files: `src/lib/masters.js`, `src/sample/masters.js`.
- Modified: `src/lib/lib.js` (`createLib` signature + wiring — resolves theme, invokes both factories once, merges, exposes `lib.masters`/`lib.masterDefinitions`), `bin/forge.js`, `src/tools/browser/compile.js` (both orchestrators gain an `applyMasters(pptx, lib.masterDefinitions)` call and optional `masters.js` loading — `compile.js`'s `compileDeck()` gains a `masters` parameter alongside its existing `theme` parameter), `bin/create.js` (copies the new sample file), `INSTRUCTIONS.md`, `COMPONENTS.md`, `lib.d.ts` (new types for the factory shape, `lib.masters`, `lib.masterDefinitions`).
- Also modified for browser-tool parity: `scripts/build-browser.js` (embeds `src/sample/masters.js` as `mastersPlaceholder`, mirroring the existing `themePlaceholder`), `src/tools/browser/app.js`/`index.html` (sidebar, toolbar, Reset, zip export/import, workspace persistence, and Move/Copy-exclusion treatment for the new permanent `masters.js` entry, mirroring `theme.js` throughout).
- No changes to `frame.js`, `primitives.js`, `components.js`, `tables.js`, `layout.js`.
- `theme.js` is unchanged. Masters are deliberately not folded into `defaultTheme`/`deepMerge`: `deepMerge` treats arrays as opaque values to fully replace (`isPlain` excludes arrays), so a `masters` array living inside `defaultTheme` would be wholesale-replaced by any workspace override touching it, silently breaking the by-title merge this change requires. Keeping `masters.js` separate, with its own `mergeMastersByTitle`, avoids that.
- The factory function is invoked exactly once per compile, inside `createLib`, before `pptx` exists — `applyMasters` (which runs later, after `pptx` is constructed) never calls it again, only registers the already-merged plain array. This keeps `applyMasters` free of any `theme` dependency.
- No breaking change to existing workspaces: `masters.js` is optional at the compile-pipeline level, and slide files that never reference `masterName` are unaffected.
